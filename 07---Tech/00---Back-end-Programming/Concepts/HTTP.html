Stateless

the server is stateless, it means client for each actions - each time needs to request to the web server

benefits

  1. Simplcity
  2. scalability

because that HTTP is stateless, then developers often imlements state managment technichs such as:

  1. coockies
  2. session
  3. tokens for things like user logins, or shoping, etc.

It’s very important that client always initial the request.

Before sending anything, client and server needs to make a connection, they will provide it with TCP (Transfer Layer Protocol) that will initial the connection between 2 devices.

HTTP Versions

Enable multiple request and response over the same connectino Caching

Headers

Headers are key-value pair of different perramiter, but why we use beaders?:

Headers are actual meta data values that we send to server or recieve from server to make a proper connection. Also headers have more use cases.

example of useful header keys/values:

Request headers

  • User-Agent: identifies what kind of client is that
  • Authorization: to identifies the user (use cases like tokens)
  • Accept: which provide information what kind of content we are expect

General headers

we use this type of header both in req and resp

  • Date: date of the message
  • Cache-Control: Caching mechanism
  • Connection: connection information like keep it alive or close it.

Representation headers

Usually specifies the way to represent the data

  • Content-Type: the actual type of content (json, image, HTML)
  • Content-Length: The length of the whole size of the resouce packet in bytes
  • Content-Encoding: which kind of encoding does data has (like gzip, diffler)
  • ETag: unique Identify which mostly uses per cache

Security Headers

Enhance the security of request/response by controlling the behavior of caching and, etc.

  • Strick-Transport-Security (HSTS): Insuring that client only communicate with server, or https preventing protocol downgrade.
  • Content-Security-Policy (CSP): strict which content like js,image,html can be loaded
  • X-Frame-Options: Prevents the page to be embedded in iframe
  • X-Content-Type-Options: Prevent the browser to quess the mimetype
  • Set-Cookie: Secure cookies that are accesiable to javascript, and ensuring they are sending over the HTTPS

Headers can be used in more ares because of:

  1. Extensibility: when we use headers, we can use/add features without modifies the underlying HTTP/HTTP protocol, some of them can even be custom that made by other devs (X-* headers)
  2. Remove Control: We can use header to actually control the behavior of server by providing some options inside the header, for example we can authenticate the user or server can set a cache mechanism to cache the data for client browser, or we can set our own custom header for more extensible functions.

HTTP Methods

Http method are exists to represent different kinds of actions by client, like browser or api consumer. For example If we want to get the name of the user, we can set the Intent of our request, then we can functions differently than for example modify it.

Http methods are:

  1. GET: we use it to get some resources from the server
  2. POST: We use it to put some resources to the server, it has the body to set data from the client
  3. PUT: Update some resources from the server but it should replace whatever we provide as body to the target resource (Complete Replacement)
  4. PATCH: Update some resources from the server but it should only patch (modify) the part of the resource that we provided as body (Modify only needed parts)
  5. DELETE: We use to delete a resource from the server.
  6. OPTIONS (CORS): it used to fetch the capability of the server for Cross Origin Requests (CORS), it controls how browser should communicate the the server resources from different host addresses.

Note that Host is used for the target api that we want to request. And we use Origin is used for our own local domain (or maybe a domain from a server).

Response Code

they will return the final state or situation of functioned request from the server as a code, this code perfectly specifies what was the exact result of the request.

This code is standard and works across all platform, languages, systems, etc.

On High level response codes composed from:

  1. Information: 1xx
  2. Success: 2xx
  3. Redirection: 3xx
  4. Client Error: 4xx
  5. Server Error: 5xx

for example 200 is OK message that means request was succsessful and completed properly by the server.

Or 201 means resource has been fulfill in the reason of Creation a resource (F.g POST Method) or 204 means request has been fulfill in the reason of accepting content, but there is nothing to return by the server itself (Nothing to return).

301 means move permanently, that means the requested resource has been moved to a new URL, and request should be in that new URL. (F.g backwards compatibility)

302 means move temporarly, that means the request resource has been temporarely moves to a new URL, but the future request will continue uses the main URL. (F.g for testing things or running temp events like campains)

304 means the resource has not been modified since the last request (F.g GET Method, Caching)

400 means bad request, we use to when the data from client is not correct and server needs a better format of data.

401 means Unauthorized, we use this to say you are not authorized or failed at that level (F.g server is using JWT, but client didn’t provide that or it is expired)

403 means Forbidden, we use this when the client tried to access a resource but they don’t have the permission for that (like when we are filtered from other sites xd)

404 means Resource not Found, we use this when client wants some resources that are not available in the server.

405 means Method is not allowed, we use this to give information to client that this method is not allowed for this server or for example for this route.

409 means conflict, we use this for miltiple reason, but for example we can return this code when use wants to add an resource that is already available to the server, and server can say that there is some conflict with available resource here.

429 means too many request, we use this to allow limited amount of requests and if they are succeeds out of that, we return this code (F.g Rate limit systems)

500 means internal server error, we use this to give information about some internal server errors, this is not related to clients and it’s only for internal components of the server. (F.g Something unexpected happend)

501 means not implemented, we use this when the server doesn’t support the HTTP method or any functionality, but it plants to implement it soon.

502 means bad gateway, Usually proxy server use this, they use it to say upstream proxy server returns an invalid response.

503 means service is not available

504 means gateway timeout, for example when we request to a server that uses reverse proxy in front and that reverse proxy cannot access the internal local server (Main web server), after some time (timeout) that reverse proxy can return this code.

Caching

We use this mechanism to recognize the re-use of the resource and cache them then, we can more quckily access the resource.

we usually control cache with HTTP headers, such as:

Cache-Control header

means: you should maintain the cache for this resource for N seconds

Cache-Control: max-age=30, public

In this example, client should maintain the cache for 30 seconds.

ETag header

they are usually hashes of the response message

Etag: 3141

Last-Modified Header

with this header method, we can see what was the last time that resource modifies

Last-Modified: Fri, 28 Sep 2024 13:48:33 GMT

what Usually happens here is that server will send those 3 headers that we talked about to the server, then the next time client request the resources, client will provide those header values with a new key names such as:

  1. If-None-Mache (same as Etag)
  2. If-Modified-Since (Same as Last-Modified)

To the server, and then server will check those with its own values of those headers, if they are same then return the 403 code (Not Modified) and if they are not same, then return the newly created resource to the client. it’s also noticeable that server will add another header called Cache-Control that we talked about, that means please do the new request after this N seconds, until that cache the current data.
